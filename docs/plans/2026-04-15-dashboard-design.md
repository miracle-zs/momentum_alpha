# Live Dashboard Design

**Date:** 2026-04-15

**Goal:** Add a small read-only web dashboard for the running strategy so operators can inspect health, current state, and recent audit events without logging into multiple shell commands.

## Scope

This first version is intentionally narrow:

- single read-only page
- no login system
- no order controls
- no database dependency
- no historical charting

The page will read the same local runtime artifacts the system already produces:

- `var/state.json`
- `var/audit.jsonl`
- health status derived from the existing `healthcheck` logic

## Recommended Approach

Build a minimal Python HTTP server inside the existing project and serve one server-rendered HTML page plus a tiny JSON endpoint.

Why this approach:

- lowest deployment friction in the current codebase
- reuses the existing Python runtime and file-based state
- avoids adding a frontend framework before the data model is stable
- keeps the first version easy to debug on the server

## Page Layout

### 1. Health Summary

Show the overall runtime verdict and the four file-based checks already exposed by `healthcheck`:

- `state_file`
- `poll_log`
- `user_stream_log`
- `audit_log`

Each row should show status plus the freshness message already generated by the health logic.

### 2. Runtime Snapshot

Show a compact overview derived from `state.json` and recent audit events:

- current `previous_leader_symbol`
- current open position count
- tracked order-status count
- last poll worker start
- last user-stream worker start
- latest tick timestamp

### 3. Recent Audit Feed

Show the most recent structured events from `audit.jsonl`, with newest first.

Priority event types:

- `poll_worker_start`
- `poll_tick`
- `tick_result`
- `broker_submit`
- `stop_replacements`
- `user_stream_worker_start`
- `user_stream_event`

## Data Flow

1. Web handler reads `state.json`.
2. Web handler reads the latest N lines from `audit.jsonl`.
3. Web handler calls the existing runtime health builder.
4. Web handler renders a single HTML page from these three sources.

No background cache is needed for v1.

## Error Handling

- Missing files should render explicit warning cards rather than crash the page.
- Invalid JSON in `state.json` or malformed audit rows should be surfaced as dashboard warnings.
- The HTTP handler should return `200` with partial data whenever possible, reserving `500` for unexpected application errors.

## Testing

Add tests for:

- dashboard data aggregation from sample `state.json` and `audit.jsonl`
- HTML route rendering with missing files
- JSON route rendering with healthy data

## Deployment

First version should run locally on the server only. If it proves useful, later work can add:

- a dedicated systemd service
- reverse proxy
- authentication
- richer historical views backed by SQLite/PostgreSQL
